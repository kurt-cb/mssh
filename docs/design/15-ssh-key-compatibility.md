# 15. SSH Key Compatibility and Migration

The single most important fact about SSH in 2026 is that almost nobody uses certificates. SSH public keys, on the other hand, are everywhere: in every developer's `~/.ssh/`, in every GitHub account, in every cloud provider's user dashboard, in every CI runner's secrets, in `authorized_keys` files on millions of servers, in Ansible inventories, in Kubernetes secrets, in backup scripts, in embedded device images. An mssh design that requires users to abandon all of this and adopt a new credential model is a design that nobody adopts.

This document defines how mssh works with the existing SSH public-key world. The short version: **the cert is a CA-attested wrapper around the user's existing public key, not a replacement for it.** Users keep their keys. Their existing tooling keeps working. The cert adds policy, audit, and lifecycle on top — for servers that ask for it.

## The model

A user's identity in mssh is anchored to a public key — typically the same key they already have. The cert binds that public key to a CA-attested identity with policy extensions (source binding, time windows, channel grants, etc.). The private key never leaves the user's machine (or smartcard). The CA never sees it.

Three states a user can be in:

1. **Unenrolled.** The user has an SSH public key but no mssh cert. They can authenticate to mssh servers if and only if those servers accept bare public-key auth (the operator's choice).
2. **Enrolled.** The user has an SSH public key *and* an mssh cert wrapping it. Same private key, same SSH agent, same workflows. The cert carries policy. When connecting to an mssh-aware server, the cert is presented and policy is enforced. When connecting to a classic SSH server, the cert is irrelevant; the public key works as it always did.
3. **CA-generated.** The user's keypair was generated by enrollment tooling (or the smartcard's onboard generator) at the time of CA enrollment. Functionally identical to "enrolled" — there is no protocol-level distinction between a CA-generated key and a pre-existing key the CA later certified.

The cert is metadata. The key is the credential.

## Enrolling an existing SSH key

The enrollment workflow:

1. The user already has `~/.ssh/id_ed25519` (or `id_rsa`, or `id_ecdsa`). They have used this key for years.
2. The user runs `mssh-enroll --key ~/.ssh/id_ed25519.pub` (or the equivalent web-based enrollment).
3. The tool generates a PKCS#10 CSR carrying the public key portion of the user's existing key, signed by the corresponding private key (proof of possession).
4. The CSR is submitted to the CA. The CA prompts for whatever 2FA the issuance policy requires.
5. The CA issues a cert binding the user's public key to their identity, with whatever extensions the policy grants.
6. The cert is delivered back to the user and stored — typically at `~/.ssh/<keyname>-cert.pem` or in the SSH agent.

The user has not generated a new keypair. Their private key has not moved. Their public key, in identical form, is what GitHub sees, what their existing `authorized_keys` entries reference, and what the cert now wraps. They have added metadata, not changed identity.

For users without an existing key, `mssh-enroll --generate` generates a new keypair locally (defaulting to Ed25519), uses it to construct the CSR, and proceeds as above. The keypair is still the user's — generated on their machine, never sent to the CA.

## The key inside the cert: bit-for-bit identical

A point worth being explicit about, because the implication is operationally important: **the public key inside the mssh cert is the same bytes as the user's SSH public key.** Not "compatible" or "interoperable" — *identical*, at the level of the curve point (Ed25519, ECDSA) or modulus/exponent pair (RSA). The cert is a wrapping; the key is the key.

A few corollaries:

- An Ed25519 SSH public key (`ssh-ed25519 AAAAC3Nz...`) decodes to 32 bytes representing a curve point. The same key in an X.509 cert is the same 32 bytes inside a SubjectPublicKeyInfo SEQUENCE. Different encoding envelope; identical key.
- The signature operations are identical too. An Ed25519 signature produced by the user's SSH private key verifies against the cert's subject public key, and against the bare SSH public key, with the same result. The cryptographic verification doesn't care which envelope the key arrived in.
- A user's `~/.ssh/id_ed25519.pub` and their mssh cert refer to the same identity at the cryptographic level. The cert adds CA attestation and policy; nothing else changes.

This means format conversion is a real and useful operational primitive. Reference implementations provide:

```
# Recover the SSH-format public key from an mssh cert
mssh-keygen --cert-to-pubkey ~/.ssh/id_ed25519-cert.pem > id_ed25519.pub

# Verify they match
diff <(mssh-keygen --cert-to-pubkey cert.pem) ~/.ssh/id_ed25519.pub
# (no output — they are identical)

# Construct an authorized_keys entry from the cert
mssh-keygen --cert-to-pubkey cert.pem --format authorized-keys
# Output: ssh-ed25519 AAAAC3Nz... alice@example.com
```

Why this matters in practice:

- **Migration to mssh.** The user's existing `authorized_keys` entries across all classic SSH servers continue to work, unchanged, because they reference the same public key the new mssh cert wraps. No rewriting required.
- **Coexistence during migration.** A user can be enrolled in mssh (has a cert) while still being authorized on classic servers (still listed in `authorized_keys` files). Both use the same private key. Both authenticate the same person.
- **Forward portability.** If a deployment later decides to move away from mssh (unlikely but possible), the cert's contents can be discarded and the public key extracted; the user's identity continues to exist in the classic SSH world without re-keying.
- **CA migration.** If an organization changes CAs, the new CA issues a new cert wrapping the user's existing key. The user keeps their key; only the wrapping changes.

The cert is metadata. The key is the credential. This is the same model TLS uses (a TLS cert wraps a public key; the key is what does the actual cryptographic work; the cert is what gets validated for trust). mssh follows the established pattern.

## How the key was generated doesn't affect cert validity

A common question during enrollment: "I generated my key with `ssh-keygen` years ago. Is it OK to use that for mssh, or should I generate a fresh one with `openssl`?"

The answer is that the *generator* doesn't affect the resulting key's security. What matters is:

- The algorithm and key size meet mssh's algorithm requirements (Ed25519, ECDSA P-256/P-384, or RSA 2048/3072/4096 — see `02-cert-profile.md`).
- The entropy source at generation time was a CSPRNG seeded from real randomness. On modern systems, this is the OS-provided random source (`/dev/urandom` or `getrandom(2)` on Linux, `BCryptGenRandom` on Windows, `arc4random` on BSD/macOS), which both `ssh-keygen` and `openssl` use.
- The private key has been stored securely since generation (on a smartcard, in an agent's memory, in a file with appropriate permissions and ideally a passphrase).

If those hold, the key is fine for enrollment regardless of which tool generated it. `ssh-keygen` and `openssl genpkey` produce mathematically equivalent keys for the same algorithm.

Failure modes that *would* warrant generating a fresh key:

- The original key was generated on a system with a broken CSPRNG (the 2008 Debian OpenSSL bug, certain early VM cloning scenarios, embedded devices with no entropy source at first boot).
- The private key may have been exposed (laptop loss, suspected compromise, unencrypted backup that got published).
- The key is RSA-1024 or weaker, below the profile's permitted size.
- The key is in a deprecated algorithm (DSA, anything with a Diffie-Hellman group of less than 2048 bits).

In any of these cases, the user generates a new key as part of enrollment. The CA's enrollment flow can detect weak keys and refuse to issue, prompting the user to regenerate.

## Wire-level compatibility

The on-wire change for SSH-key compatibility is smaller than `04-handshake-and-hop-chain.md` initially implied. mssh authentication uses the standard SSH `publickey` userauth method framing, with an extended algorithm string that signals "this is a cert, not a bare public key."

### Classic public-key auth (RFC 4252 § 7)

```
byte      SSH_MSG_USERAUTH_REQUEST
string    user name
string    service name
string    "publickey"
boolean   TRUE                         ; has signature
string    public key algorithm name    ; e.g. "ssh-ed25519"
string    public key blob              ; SSH wire format
string    signature                    ; over session id || preceding fields
```

### mssh cert auth — same shape, different algorithm name

```
byte      SSH_MSG_USERAUTH_REQUEST
string    user name
string    service name
string    "publickey"
boolean   TRUE
string    public key algorithm name    ; e.g. "mssh-cert-ed25519-v01"
string    public key blob              ; cert + chain, see below
string    signature                    ; over session id || preceding fields
```

The algorithm name distinguishes the two cases. A server that does not recognize `mssh-cert-ed25519-v01` returns the normal "unsupported method" failure, just as it would for any unknown algorithm. A server that does recognize it parses the public key blob as a cert and chain rather than as a raw key.

### The mssh cert public key blob

```
string    "mssh-cert-v01"             ; format identifier
string    cert serial number          ; hex, for logging
string    cert DER                    ; the X.509 leaf
uint32    chain length N
   N times:
   string  intermediate DER           ; ordered leaf-issuer first
uint32    flags                       ; rotation-requested, hop-chain-follows, etc.
```

The signature is computed over the standard session id and request fields, using the private key corresponding to the cert's subject public key. The signature algorithm is the algorithm of the cert's subject key (ed25519, ecdsa-sha2-nistp256, rsa-sha2-256, etc.) — the same algorithms classic SSH uses, because the keys are the same keys.

This is the entire wire-level change for mssh user authentication. Hop chain attestation, server identity verification, and the other mssh-specific messages remain as defined in earlier docs, but the user auth itself rides on the standard SSH `publickey` method with extended algorithm names.

### What this gives us

- A single client can present either a bare public key (classic auth) or a cert (mssh auth) to the same server, and the server makes the policy decision based on what it received.
- Existing SSH client libraries can be extended to mssh by adding the new algorithm names and the cert parsing — no changes to the userauth state machine.
- Existing SSH servers that have not been updated to mssh see `mssh-cert-ed25519-v01` as an unknown algorithm and fall through to whatever auth methods they do support. The client can retry with bare public-key auth if needed.

## Server configuration: the three modes

Operators choose what their servers accept on a per-host or per-deployment basis.

### `mssh-only`

The server accepts only `mssh-cert-*` algorithm names. Bare public-key auth is refused. This is the strictest mode and the recommended posture for production servers once the user base has been enrolled.

```
# sshd_config
AuthenticationMode mssh-only
```

### `mssh-preferred`

The server accepts both cert-bearing and bare public-key authentication. Cert-bearing connections get full mssh policy enforcement (source binding, time windows, channel grants). Bare public-key connections get baseline access — typically a shell with no channel plugins available and possibly with extra logging.

```
AuthenticationMode mssh-preferred

# What bare public-key auth gets when accepted:
LegacyClientAccess {
    AuthorizedKeysFile %h/.ssh/authorized_keys
    PermitChannelTypes none
    LogAttribute auth-mode=legacy
    Until 2027-01-01
    Reason "Phasing in cert-based auth across user base"
}
```

The `Until` field is required, the same forcing function used for legacy server-side compatibility. Operators cannot leave bare public-key access enabled indefinitely without revisiting and re-justifying the decision.

### `classic-only`

The server accepts only bare public-key auth. mssh certs are ignored or refused. This mode is intended for testing and for environments that have not yet begun the migration. It carries the full security profile of classic SSH (which is to say, the profile this entire project exists to improve upon).

```
AuthenticationMode classic-only
```

`classic-only` exists for completeness and for testing. It is not the recommended deployment.

## What changes for the user

Materially, very little.

**A user enrolled in mssh:**

- Continues to use their existing SSH key.
- Continues to use `ssh-agent` and `ssh-add` as before.
- Continues to have GitHub access via the same key.
- Connects to mssh-aware servers and gets full policy enforcement.
- Connects to classic SSH servers and authenticates normally (the cert is just metadata the classic server doesn't read).
- Rotates their cert daily or weekly (via the CA), but the underlying key stays the same.

**A user not yet enrolled:**

- Uses SSH as before.
- Can connect to mssh servers configured in `mssh-preferred` mode using their existing key.
- Cannot connect to `mssh-only` servers until they enroll.
- Has no source binding, time windows, channel grants, or audit beyond the server's local logs.

The migration path for an organization is: deploy mssh sshd everywhere in `mssh-preferred` mode → enroll users gradually → tighten servers to `mssh-only` once enrollment is sufficient → remove the `LegacyClientAccess` configurations.

## What changes for operators

A new operational responsibility: track which users are enrolled. The CA's audit log records every issuance, so this is mechanical:

```
mssh-ca enrollment-status --org example.com
  Enrolled (last 30 days):     487 users
  Enrolled but inactive:       12 users
  Never enrolled:              43 users
  Total:                       542 users
```

The "never enrolled" list is the migration backlog. Operators can apply gentle pressure (deadlines, automated emails, server-side warnings) without breaking anyone's access.

## Cert rotation with a long-lived key

This is the subtle part of the model. The user's underlying SSH key may live for years. The cert wrapping it lives for hours or days. How does this work?

The answer is that the cert wrapping is purely a *binding* — it asserts "the CA confirms that this public key belongs to alice@example.com with these policies, attested at time T, valid until T+8h." When the cert expires, the CA issues a new cert binding the *same* public key to the *same* identity with the *same* (or refreshed) policies. The user's keypair is untouched.

This is exactly how X.509 in TLS works. A web server's TLS cert is renewed every 90 days; the server's underlying keypair is typically refreshed less often (sometimes never, though best practice is to refresh on major rotations). The cert is the binding, not the key.

For mssh users this means:

- Their `~/.ssh/id_ed25519` lives as long as they keep it. Years, possibly.
- Their `~/.ssh/id_ed25519-cert.pem` is replaced every issuance cycle.
- 2FA happens at cert issuance time, not at every connection.
- Compromise of the cert without compromise of the key means: the cert's policy is enforceable until it expires, and a new cert can be issued binding the same key. Compromise of the key means: the user generates a new key and re-enrolls.

Key rotation and cert rotation are independent operations on independent schedules. This is good — they have different risk profiles and different operational costs.

## When to rotate the underlying key

The cert wrapping doesn't change the fundamental advice on SSH key rotation, but it does provide better visibility for when rotation has happened. Recommended triggers for generating a fresh keypair:

- Smartcard or device loss (mandatory; old key is potentially compromised).
- Suspected machine compromise (mandatory).
- Move to a stronger algorithm (e.g., upgrading from RSA-2048 to Ed25519 or to a post-quantum algorithm).
- Periodic rotation per organizational policy (recommended, not mandatory in mssh).

When the key rotates, the user enrolls again. Their old cert is revoked at the CA (so servers stop accepting it on its next validation pass). Their new key gets a fresh cert. Their `authorized_keys` entries on classic-only servers need to be updated; the CA can maintain a list and trigger ansible/salt/equivalent.

## The compatibility ceiling

A user authenticating via bare public-key to an `mssh-preferred` server gets baseline access only. Specifically, they do NOT get:

- Source-address binding (no enforcement).
- Time windows (no enforcement).
- Channel plugin access (cert grants required).
- Fresh 2FA enforcement (no cert means no 2FA evidence).
- Hop chain participation (hops require valid certs).
- Session rotation (rotation requires cert metadata).
- Useful audit beyond "user X connected with key fingerprint Y at time Z."

Operators who want any of these things must require certs (`mssh-only`). The `mssh-preferred` mode exists to enable enrollment, not to be the long-term destination.

The threat model implication is that during the migration period, an organization's security posture is no better than classic SSH for unenrolled users, and better than classic SSH for enrolled users. The gradient is positive — every user enrolled is a user with stronger access — but the curve only reaches its full slope when `mssh-only` is enforced.

This is the price of adoptability. The alternative is a flag day, which doesn't work.

## Implementation requirements

For mssh servers:

- MUST recognize the `mssh-cert-*` algorithm name family.
- MUST validate the cert chain and signature when this algorithm is presented.
- MAY accept bare public-key auth per configured policy.
- MUST log the authentication mode (cert vs bare key) in connection records.
- MUST refuse to silently downgrade — a client presenting a cert that fails validation does not automatically retry as bare public-key.

For mssh clients:

- MUST be able to present either bare public-key auth or cert-based auth, controlled by the user's local configuration or by detection of available certs.
- SHOULD present cert-based auth by default when a cert is available for the target server.
- MUST gracefully handle the case where a server does not recognize `mssh-cert-*` algorithms (fall through to bare public-key or fail clearly).
- MUST NOT send certs to servers known to be classic-only unless explicitly configured (privacy: the cert carries the user's identity and policy details).

For CAs:

- MUST accept CSRs containing arbitrary supported public keys, not requiring CA-generated keypairs.
- MUST verify proof-of-possession on every CSR.
- MAY support an "existing key import" workflow that walks the user through enrollment of an SSH key file.

## Migration playbook for an organization

A realistic timeline for a 500-user organization with classic SSH:

**Week 1-2: CA setup.**
Set up the CA (embedded for proof-of-concept, external for production scale). Define the issuance policy. Test with a small group of administrators.

**Week 3-4: Server pilot.**
Deploy mssh sshd to a pilot set of servers in `mssh-preferred` mode. Existing access continues to work; cert-based access is available for users who want it.

**Week 5-12: User enrollment campaign.**
Roll out enrollment tooling. Communicate the change. Provide self-service enrollment with clear instructions. Track enrollment rate weekly. Address blockers (users with non-standard key types, users with hardware tokens, users with `authorized_keys` setups that conflict with the migration).

**Month 4-6: Server-side tightening.**
On a per-server basis, switch from `mssh-preferred` to `mssh-only` once enrolled users represent enough coverage. Coordinate with affected teams. Servers that aren't ready stay in `mssh-preferred`.

**Month 7-12: Sunset bare public-key.**
By end of year 1, most servers are `mssh-only`. The remaining `mssh-preferred` servers have explicit `Until` dates and documented reasons. The migration is functionally complete; cleanup continues.

This is realistic. It is not a six-week sprint. The reward at the end is the full mssh security posture across the fleet — but the path there respects that nobody can move everyone at once.

## What this does not change

To be explicit about what stays:

- The hop chain integrity model. Hops still need certs (no "bare-key hops").
- The channel plugin framework. Channels still require cert grants.
- The DNSSEC requirement.
- The threat model for fully-enrolled deployments.
- The protocol's rejection of password authentication. Passwords are still gone.

What changes is that mssh now coexists with the world as it actually is, rather than requiring the world to change before mssh can be useful.
